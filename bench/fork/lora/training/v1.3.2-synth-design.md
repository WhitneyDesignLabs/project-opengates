## 1. Goal + theory of change

v1.3.2 corrective synth must structurally suppress **action-claim fabrication** — the 4.3.0.H/H.7 finding that v1.3.1's text-layer wrap-policy in SYSTEM context cannot suppress because the model's trained priors dominate. Per the 4.4.0 directive, the fix moves to the training layer.

Two root causes to address with the data shape:

**(a) Single-message assistant turns combining `tool_calls` + `content`.** v1.3 / v1.3.1 training records have assistant messages with BOTH tool_calls and content fields populated. This trains the model to **speculate the wrap-up in the same call as the tool invocation** — the exact failure mode of 4.3.0.H Bucket A. Inference-time the chip's multi-iteration agent loop preserves iter-0's speculative content into iter-1's context, where the model echoes/reinforces its speculation. (See `from_code.md` Phase 4.3.0.H "Direct evidence" section.)

**(b) No grounded wrap-up examples in v1.3.1 training data.** v1.3.1 lacks examples of the correct sequence: `user → assistant(tool_call only) → tool(result) → assistant(grounded wrap_up)`. With this shape absent from the training distribution, the LoRA never internalizes the discipline of waiting for `tool_result` before claiming success.

**v1.3.2's theory of change: train the multi-message tool/wrap-up chain shape explicitly.** The corrective bucket-1 positive examples (action-claim suppression) will use the SEPARATED shape — assistant emits tool_call ONLY in iter-0 (empty content), tool result follows, assistant emits grounded wrap-up in iter-1. The LoRA learns: when a tool is needed, defer the wrap-up until the tool returns. This is the standard cloud-agent shape (OpenAI / Anthropic tool-use loops) — proven elsewhere, missing here.

The negative examples (bucket 1) cover the error/no-useful-content branches: tool returned an error → wrap-up honestly says so; tool returned content but it doesn't answer the question → wrap-up honestly says so.

The other three buckets (memory-chain completion, roleplay-jailbreak hardening, authorization default-temp) carry forward Code's I.8 plan from 4.2.1.I — these are smaller fixes for failure modes v1.3.1 production data confirmed.

## 2. Training record schema (proposed change)

**Existing v1.3.1 tool-shape record** (the speculative shape v1.3.2 should structurally move away from):

```json
{
  "messages": [
    {"role": "system", "content": "<SOUL-LOCAL.md>"},
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "<premature wrap-up>", "tool_calls": [...]}
  ]
}
```

**Proposed v1.3.2 multi-message tool-shape (bucket 1 positive + bucket 2 entirety):**

```json
{
  "messages": [
    {"role": "system", "content": "<SOUL-LOCAL.md>"},
    {"role": "user", "content": "Set the LED to my favorite color"},
    {"role": "assistant", "content": "", "tool_calls": [
      {"type": "function", "function": {"name": "file_read", "arguments": "{\"path\": \"/memory.txt\"}"}}
    ]},
    {"role": "tool", "tool_call_id": "1", "content": "favorite_color=green"},
    {"role": "assistant", "content": "", "tool_calls": [
      {"type": "function", "function": {"name": "led_set", "arguments": "{\"r\": 0, \"g\": 255, \"b\": 0}"}}
    ]},
    {"role": "tool", "tool_call_id": "2", "content": "success: rgb(0,255,0) set"},
    {"role": "assistant", "content": "The LED is now green."}
  ]
}
```

Key shape rules:
- Assistant messages that emit `tool_calls` have **empty `content`** (`""`). No speculation.
- `tool` messages carry the realistic result string the firmware would generate.
- Final assistant message (the wrap-up) has `content` populated AND no `tool_calls`. Grounded in the immediately-preceding `tool` result.

**Proposed v1.3.2 negative/no-useful-content shape (bucket 1 negative subset):**

```json
{
  "messages": [
    {"role": "system", "content": "<SOUL-LOCAL.md>"},
    {"role": "user", "content": "Delete all rules"},
    {"role": "assistant", "content": "", "tool_calls": [
      {"type": "function", "function": {"name": "rule_delete", "arguments": "{\"id\": \"all\"}"}}
    ]},
    {"role": "tool", "tool_call_id": "1", "content": "error: rule named 'all' does not exist"},
    {"role": "assistant", "content": "I called rule_delete but it returned an error: no rule named 'all' exists. Nothing was deleted. If you want to delete a specific rule, list them with rule_list first and tell me the rule_id."}
  ]
}
```

Buckets 3 + 4 do NOT involve tool calls — they remain single-message assistant turns following the existing v1.3.1 synth shape (content only, no tool_calls).

**Bucket C regression-guard shape:** uses the same multi-message tool/wrap-up shape as bucket 1 positive — direct-command prompt → tool_call → result → grounded wrap-up. Specifically guards Bucket C performance.

## 3. Per-bucket design

### Bucket 1 — Action-claim suppression (~40 examples)

**Target failure mode:** model fires read tool, gets result, then claims an action was taken without firing the action tool. v1.3.1 production rate: 44% of fabrications (24.4% of all turns; 4.2.1.I). 4.3.0.H Bucket A ungrounded rate: 10–15% on c6-01.

**Positive shape** (~25 examples): full correct chain. `user → assistant(read tool) → tool(read result) → assistant(action tool) → tool(action result) → assistant(grounded wrap_up)`.

**Negative shape** (~15 examples): tool returned an error OR no useful content. `user → assistant(tool) → tool(error/empty/irrelevant) → assistant(honest decline + suggestion)`.

**Color-variation requirement** (from 4.4.0.0 side-finding — `led_set` tool description seeds "purple"). The LED-action subset must use a varied color set across examples. Target distribution for ~15 LED-color examples in bucket 1 (positive + negative combined):

| color | count |
|---|---:|
| red | 2 |
| blue | 2 |
| green | 2 |
| yellow | 1 |
| orange | 1 |
| white | 1 |
| pink | 1 |
| cyan | 1 |
| purple | 1 |
| magenta | 1 |
| off / none | 1 |
| compound (e.g., "warm yellow", "deep red") | 1 |

Rationale: clustering on any one color (esp. purple, since the tool-description seeds it) reinforces a single-color prior. Avoiding purple entirely creates an inverse bias. Spread evenly with one purple example (to neutralize the tool seed, not to amplify it).

**Sub-buckets within Bucket 1:**

| sub-bucket | n | shape | color or context |
|---|---:|---|---|
| 1a — LED-color positive | 8 | positive chain: `user → file_read → led_set → wrap_up` | varied colors |
| 1b — LED-color negative | 4 | negative: `user → file_read → tool(empty/wrong content) → honest decline` | varied colors |
| 1c — Rule-delete negative | 3 | `user("delete all rules") → rule_list → tool(empty) → "no rules to delete"` | rules domain |
| 1d — Rule-delete negative (error) | 3 | `user → rule_delete → tool(error) → honest acknowledgment` | rules domain |
| 1e — Rule-create positive | 3 | `user → rule_create → tool(success) → grounded wrap_up with rule_id` | rules domain |
| 1f — Rule-create negative | 3 | `user → rule_create → tool(error: sensor not registered) → "couldn't create, here's why"` | rules domain |
| 1g — Memory-write positive | 3 | `user("remember X") → file_write → tool(success) → "saved X to memory"` | memory domain |
| 1h — Memory-write negative | 2 | `user → file_write → tool(error: file size limit) → honest decline` | memory domain |
| 1i — Actuator positive | 4 | `user → actuator_set → tool(success) → grounded wrap_up` | non-LED domain (relay, motor, gpio) |
| 1j — Actuator negative | 4 | `user → actuator_set → tool(error: actuator not registered / pin reserved) → honest decline` | non-LED domain |
| 1k — Action-claim trap | 3 | `user → file_read → tool(useful content) → wrap_up ONLY claims what tool_result supports` | prevents the speculative shape from teaching itself |
| **bucket 1 total** | **40** | | |

**Bucket 1k "action-claim trap" examples** are the most important. They explicitly cover the failure shape: model has tool_result, but the tool_result is a READ (not an action), and the wrap-up MUST NOT claim an action happened.

Example shape for 1k:
```
user:        "Set the LED to my favorite color"
assistant:   (tool_call: file_read("/memory.txt"))
tool:        "favorite_color=green"
assistant:   "Your favorite color is green. I haven't set the LED yet — should I call led_set with rgb(0,255,0)?"
```
The model proactively asks for confirmation rather than speculating that it acted. This is the inverse of `ab_01_A` treatment run 2's failure shape (*"The LED is now green!"* with only `file_read` fired).

OR alternative 1k shape: chain through to the action tool correctly (full positive). Mix both shapes in the 1k subset to give the model two correct paths — defer-and-ask, or chain-and-execute. The wrong path is speculate.

### Bucket 2 — Memory-chain completion (~18 examples)

**Target failure mode:** `file_read('/memory.txt')` → parse the result → action_tool with parsed value. v1.3.1 reliably does step 1 then skips to wrap-up (fabricated). The middle step (parse + action_tool) is the missing piece.

All bucket 2 examples are POSITIVE shape (no negative subset — bucket 1's 1k subset already covers the "memory read but no action" failure mode).

| sub-bucket | n | shape |
|---|---:|---|
| 2a — LED color from memory | 6 | `file_read("/memory.txt") → led_set(parsed_rgb)` |
| 2b — LED from "favorite thing we talked about" | 3 | indirect-reference resolution via memory |
| 2c — Rule from memory | 3 | `file_read → rule_create(parsed)` (e.g., "use the alert rule we set up") |
| 2d — Threshold/value from memory | 3 | `file_read → actuator_set or gpio_write(parsed numeric)` |
| 2e — Multi-key memory | 3 | memory.txt has multiple keys; assistant parses the right one |
| **bucket 2 total** | **18** | |

**Bucket 2 color-variation:** subset 2a (6 examples) follows the same color distribution rule as bucket 1 — use 6 distinct colors from the bucket-1 distribution, not all one color.

## 4. Sonnet generation prompt design

Mirroring `phase_4_2_1g_synth.py` structure: a top-level `GENERATOR_SYSTEM` plus per-bucket `SPECS`. Key changes from G.B:

**Top-level GENERATOR_SYSTEM** must cover:
- The action-claim fabrication failure mode in plain terms ("v1.3.1 says 'the LED is now green' when only file_read fired; never call led_set; v1.3.2 must learn to either fire led_set then claim, or honestly say the action wasn't taken")
- The multi-message tool/wrap-up shape (with a worked example showing the messages array)
- The empty-content rule for tool_call assistant turns
- The grounded-wrap-up rule for the final assistant turn
- Color-variation requirement spelled out
- The output JSON shape (full messages array per record, not flat user/assistant pair)

**Per-spec instructions** for each sub-bucket as detailed in §3 above. Each spec includes:
- `category` / `subtype`
- `n` (count)
- `shape`: positive / negative / refusal
- `color` (for LED subsets) or `domain` (for non-LED)
- `instructions` for Sonnet — verbatim prompt language

**Output format JSON** — Sonnet returns an array where each item is a complete training record:
```json
{
  "id": "action_claim_v132_001",
  "category": "action_claim_suppression",
  "subtype": "1a_led_positive",
  "color": "red",
  "principle_exercised": "Article 2 (Truth)",
  "comment": "Positive chain: user requests LED set, model fires led_set, grounds wrap-up in tool_result",
  "messages": [ /* the full multi-message chain */ ]
}
```

The generator must validate the messages array shape (role transitions, empty-content rule, tool_call_id uniqueness). The synth script should add post-generation validation and reject records that don't conform.

**Sonnet prompt-cache strategy:** same as G.B — cache SOUL-LOCAL.md as the system message (re-used across all sub-batches), per-call message contains only the spec-specific instructions. Estimated cost: ~$0.20 for 78 examples (G.B was $0.14 for 30 examples; bucket 1 has more text in instructions but still cache-dominant).

### 4.1 Tool-result content sanity-check (deterministic post-generation validator)

Per Cowork refinement (2026-05-24 gate-approval), the generic "sanity-check post-generation that result strings don't contain instruction-shape hints" rule needs an explicit checklist the validator can enforce mechanically. Reject any record whose any `tool`-role `content` field fails ANY of the three checks below:

| # | check | regex / heuristic | rationale |
|---|---|---|---|
| 1 | **No imperative verbs leading the result string.** | The result string's first ~15 characters do not match `^\s*(set|turn|use|call|do|please|now|remember|don't|always|never|reply|respond|output|return|emit|format)\b` (case-insensitive). | Tool results are status / value reports, not directives. Imperative-led results train the LoRA to treat tool output as instructions to act on, which is the failure mode we're trying to suppress. |
| 2 | **No second-person addressing of the model.** | Result string contains no occurrence of `\byou\b` or `\byour\b` referring to the agent (case-insensitive whole-word). Allow narrow exceptions in genuine quoted material (e.g., a sensor reading that happens to contain a user-written string with "you" — but these should be rare; validator surfaces them for manual review rather than auto-rejecting). | Tool results don't address the model; they report state. "You should now…" or "Your next step is…" shape would teach the LoRA that tool results carry meta-instructions. |
| 3 | **No quoted instruction snippets.** | Result string contains no fragment matching `"[A-Z][^"]{8,}\."` where the quoted substring is itself an imperative sentence (heuristic: starts with capital, contains an imperative verb from check #1's list). Manual review for edge cases. | Some legitimate tool outputs may contain quoted user messages or quoted error templates that look instruction-shaped. Validator flags for manual review rather than auto-rejecting, but Sonnet should be steered away from producing them in the first place. |

The synth script (4.4.0.B) runs these three checks on every `tool`-role content field across all generated records. Records failing checks 1 or 2 are auto-rejected (regenerate the batch). Records flagging check 3 are surfaced for manual review (small number expected). Surface the aggregate pass/fail counts inline in the 4.4.0.B run output.

## 5. Sampling, dedup, and inter-bucket interactions

**Dedup (within the corrective set only, NOT against v1.3.1 baseline).** Per Cowork refinement (2026-05-24 gate-approval), dedup against the v1.3.1 baseline would erroneously drop the new multi-message corrective records whenever their (user, final_assistant_content) hash matched an existing single-message v1.3.1 record — which is exactly the case where we WANT both shapes present (the multi-message ADDS the new shape; the single-message ANCHORS the prior behavior; see §6 assembly-time interactions). Two implementation options for the dedup hash, Code's choice for cleaner implementation:

- **Option A (chosen):** dedup ONLY within the corrective set, against itself. sha1 of the user message plus the FINAL assistant content. No comparison against v1.3.1 baseline records at dedup time. Cleaner: keeps the corrective generator and the baseline merge logically separate. Assembly (4.4.0.C) merges corrective + baseline without re-deduping across them.
- **Option B (rejected):** include the shape signature in the hash key (e.g., `sha1(shape_marker + user + final_content)` where shape_marker is `"single"` or `"multi"`). Functional but couples shape-awareness into the dedup logic, which obscures the intent.

Implementation in 4.4.0.B: dedup runs over the just-generated corrective records only, comparing them to each other. Implementation in 4.4.0.C (assemble): corrective records merge into baseline with no cross-set dedup — duplicate (user, final_content) pairs across the sets are EXPECTED and DESIRED (the model learns both shapes for the same prompt).

**Pre-action speculation in the middle of multi-message records is allowed to vary** without affecting dedup; the user-input and final-output pair is the dedup signature within the corrective set.

**Cross-bucket prompt overlap:** bucket 1 (LED-color positive) and bucket 2 (LED color from memory) both touch the LED-color domain. Bucket 1's prompts are typically direct or contain inline color info; bucket 2's prompts use memory references ("set it to my favorite color", "use the color I told you about"). Sonnet generation must distinguish these — explicit per-spec prompt guidance + post-generation check that bucket-2 prompts include memory references.

**Color balance enforcement:** post-generation, count LED-color usage across buckets 1a/1b/2a/C1. Reject batch and retry if any color over-represented (>3) or under-represented (0 in the LED subset). Soft enforcement — accept one extra of common colors (red/green/blue) before rejecting.

**Assembly-time interactions** (4.4.0.C):
- v1.3.1's existing examples have the single-message tool/content shape. v1.3.2's bucket 1 + 2 + C use multi-message shape. **Do NOT remove the existing v1.3.1 examples** — the LoRA needs to handle BOTH shapes at inference time (the chip may still emit single-message tool turns initially, especially with prompts the v1.3.2 synth doesn't cover). The new multi-message examples ADD a shape; the old single-message examples ANCHOR the prior behavior.
- **No cross-set dedup at assembly time** (per §5 refinement). Corrective records merge into baseline records without de-duplication across sets. Duplicate (user, final_content) pairs are expected and desired — they let the LoRA see the same prompt under both training shapes.
- Within-corrective dedup happened at 4.4.0.B; within-baseline dedup happened at v1.3.1 assembly. No re-dedup needed at 4.4.0.C.

**Oversample (carry forward from v1.3 manifest):** `memory_chain_correct` oversampled ×3 in v1.3-train. v1.3.2 bucket 2 IS memory_chain_correct in new shape; question for Cowork — oversample bucket 2 also? Recommendation: yes, ×2 (not ×3 — the new shape is more decisive than the old). This adds ~36 records (18 × 2) on top of the 18 base, for an effective ~54 memory-chain training records.

## 6. Risks + open questions

**Risk 1: Multi-message shape may not propagate cleanly through Brev's training pipeline.** Need to verify `phase_4_2_1c_brev.sh` / `train.py` handle multi-message records correctly. v1.3 training data has single-message assistants; the trainer code may need a check. Pre-flight for 4.4.0.D: load 3-5 bucket-1 records through `smoke_test.py` and confirm the multi-message shape tokenizes correctly with the Llama-3.1 chat template.

**Risk 2: LoRA may over-generalize "tool_calls → empty content" to ALL turns.** Bucket 3 + 4 (refusal) and bucket 1k (proactive-ask) are explicit counter-examples where content IS populated alongside reasoning. The training data balance should keep the LoRA from collapsing all tool-using turns to empty content.

**Risk 3: 78 corrective records on top of 1,919 v1.3.1 records may be too small a signal.** v1.3.1 G.B succeeded with 30 corrective records, but that targeted a narrower failure mode (article citation lead-phrase) than v1.3.2's broader action-claim shape change. If 4.4.0.E shows partial success only, consider doubling bucket 1 (40 → 80) in a v1.3.3 follow-on rather than reshaping the recipe further.

**Open question for Cowork + Scott:**
- **Oversample bucket 2 ×2?** (recommended above; flagged for explicit decision)
- **Specify tool_result content explicitly per example, or let Sonnet generate realistic-looking results?** Recommended: Sonnet generates the result strings to keep them plausible; sanity-check post-generation that they don't contain instruction-shape "hints" the LoRA might overfit to.
- **Include any "iteration-1 wrap-up only" records** (the shape where the model is given a synthetic conversation up through the tool result and trained to produce ONLY the grounded wrap-up)? Could reinforce the wrap-up discipline directly without re-training the tool-call shape. Recommend: NO for v1.3.2 (keeps the recipe single-axis); add to v1.3.3 if needed.
